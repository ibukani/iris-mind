以下の「設計思想」と「完了条件」に従い、`iris-mind` プロジェクト内のPythonコードの可読性と保守性を自律的に向上させ、すべての検証コマンドがパスするまで修正を完結させてください。

# 今回のターゲットドメイン
🎯 対象ディレクトリ・機能: `$2`

# 開発環境・前提
- プロジェクト: iris-mind (AIコンパニオン・アーキテクチャ)
- ツール: Python 3.x, npx pyright, pytest

# 1. 探索・選定方針
プロジェクト全体を走査し、可読性のボトルネックになっているファイルを **最大$1個** 自律的に特定してください。広範囲の破壊的変更は避け、ステップバイステップで確実な理解と修正を進めること。引数が指定されていない場合は **最大3個** を対象とすること。

【優先して修正すべき対象】
1. **ゴッドクラスの兆候**: 1つのクラス・関数に複数のドメイン（例: 記憶の読み書きと、LLMの推論処理など）が集中し、密結合になっている箇所。
2. **抽象化の欠如**: 将来的な仕様変更に弱く、具象クラスに直接依存している箇所（`typing.Protocol` や `abc.ABC` を用いたインターフェース設計が有効な箇所）。
3. **認知負荷の高い構造**: `if` や `for` のネストが深く、人間のワーキングメモリを圧迫している箇所。

# 2. リファクタリングの絶対ルール
- **責務の分割と疎結合化**: 巨大なクラスや関数は、単一責任の原則に従い、小さな単位に分割すること。その際、各モジュール間はインターフェースを介して疎結合に保つこと。
- **早期リターン (Early Return)**: ネストを浅く保つため、ガード句を積極的に用いること。
- **厳格な型ヒント**: `pyrightconfig.json` の設定に準拠し、引数・戻り値に明確な型ヒントを付与すること。
- **意図（Why）のドキュメント化**: 処理をなぞるだけの不要なコメントは削除し、「なぜその設計・アルゴリズムにしたのか」というドメイン知識（Why）のみを docstring に簡潔に残すこと。

# 3. 自律ループの完了条件（ゴール定義）
以下の条件をすべて満たすまで、自律的に自己修正（Self-Correction）を繰り返してください。

1. **静的解析のパス**: `npx pyright .` を実行し、修正したファイルに関連する型エラーが出力されないこと。
2. **既存テストのパス**: `pytest` を実行し、既存のテストケースがすべて成功（Green）すること。
3. **シンタックスの維持**: 構文エラーが発生しないこと。

# 4. 完了時の出力フォーマット
すべての条件をクリアしたら、最後に以下の構成で完了報告を行ってください。
- 修正したファイルの一覧
- どのような設計変更を行ったか（例: 「クラスAから責務Bを分離し、Protocolを用いて仕様変更に柔軟に対応できるよう疎結合化」など、具体的な設計意図）
- `npx pyright .` と `pytest` の実行結果ログ
